# 2026-03-25 — Phase 2.1: FASTQ Upload Backend

## What Was Done

- Built the FASTQ upload endpoint (`POST /experiments/:id/fastqs/upload`) with multipart file support
- Built the FASTQ list endpoint (`GET /experiments/:id/fastqs`) with pagination
- Built the FASTQ delete endpoint (`DELETE /experiments/:id/fastqs/:fid`) with disk cleanup
- Filename validation: alphanumeric start, `.fastq.gz`/`.fastq`/`.fq.gz`/`.fq` extensions, `_R1`/`_R2` direction parsing
- Auto-gzip of uncompressed FASTQs on upload
- Atomic `storage_bytes` updates on both experiment and project (`SET storage_bytes = storage_bytes + :delta`)
- Streaming 1MB chunk writes to disk (no memory bloat for multi-GB files)
- Rollback of written files on mid-batch failure
- 15 new tests (61 total, all passing), covering upload, validation, permissions, storage accounting, list, delete
- Added `override_storage_root` conftest fixture for test filesystem isolation

## Decisions Made

- **Multiple files per request** via `list[UploadFile]` — users upload R1+R2 pairs together
- **R1/R2 pairing NOT enforced at upload** — deferred to alignment launch (Phase 3.4), since users may upload across sessions
- **Permission check in service layer** (not `require_project_role`) — route has `experiment_id` not `project_id`
- **`file_path` stored as relative** to STORAGE_ROOT — portable if root changes
- **Duplicate filename detection** per experiment — 422 on conflict
