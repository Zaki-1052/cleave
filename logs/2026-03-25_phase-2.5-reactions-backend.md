# Phase 2.5 — Reactions Backend

**Date**: 2026-03-25

## What was done

- Replaced 2 stub endpoints in `reactions.py` router with 8 fully implemented endpoints (list, create, bulk create, CSV import, template download, prefix detection, update, delete)
- Created `reaction_service.py` with all business logic: CRUD, CSV parsing with CUTANA-format column mapping, boolean conversion (Yes/No → bool), template generation, FASTQ prefix auto-detection
- Extended `reaction.py` schemas: added `ReactionUpdate`, `PrefixInfo`, `CsvImportResponse`; added field validators for organism and assay_type using existing enums
- Wrote 23 integration tests covering CRUD, validation, permissions, unique constraints, bulk create, CSV import, template download, and prefix detection
- 98 total tests passing (23 new + 75 existing), zero regressions

## Decisions made

- CSV column mapping derived from `cutana/H3K4me3/H3K4me3-reactions.csv` export; `Reference_Genome` column accepted but ignored (not a reaction field)
- CSV import is all-or-nothing: if any row fails validation, entire import is rejected
- FASTQ prefix validation is soft (no error if prefix doesn't match uploaded files) since reactions may be created before FASTQs are uploaded
- Duplicated `_get_experiment_with_permission()` from `fastq_service.py` with TODO to extract to shared module (refactoring fastq_service out of scope)
- Separate endpoints for single create (returns `ReactionRead`) vs bulk/CSV create (returns `CsvImportResponse`) due to different response shapes
