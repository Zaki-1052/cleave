## What Was Done

### Job Termination
- `POST /jobs/{id}/terminate` endpoint ‚Äî terminates queued or running jobs (409 for non-terminable states)
- `TerminatedError` exception + `cancelled: Callable` parameter on `run_cmd()`/`run_piped_cmd()` ‚Äî checks DB between subprocess steps
- Sync SQLAlchemy engine in worker for termination checks (pipeline code is synchronous)
- Worker catches `TerminatedError` separately: sets status, creates notification, logs event
- Race condition handling: worker checks `termination_requested_at` before starting pipeline
- `_update_experiment_status()` and `_create_job_notification()` handle `terminated` status

## Decisions Made

- **DB polling for termination** (not signals/Redis) ‚Äî worker and API are separate processes; DB column is the simplest reliable inter-process signal for single-worker architecture
- **`retry_of_job_id` over reusing `parent_job_id`** ‚Äî avoids semantic overload; parent_job_id encodes dependency chains, retry is a different concept
- **`cancelled` callback granularity** ‚Äî checks between subprocess steps, not mid-subprocess. Response time = current subprocess step duration. Acceptable for v1.
- **Sync engine for termination checks** ‚Äî pipeline stages are synchronous (`subprocess.run`), so a 1-pool sync engine avoids async/sync mismatch
