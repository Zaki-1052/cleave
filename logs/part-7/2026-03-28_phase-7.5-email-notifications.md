# 2026-03-28 — Phase 7.5: Email Notifications + Password Reset

## What Was Done

- **Amazon SES email service**: `boto3` (sync, `asyncio.to_thread` wrapper) with Jinja2 `autoescape=True` templates. Graceful skip when SES unconfigured.
- **Job completion emails**: Wired into `worker.py` after in-app notification. Respects `always`/`on_error`/`never` user preference. Includes job name, experiment, project, duration, "View Results" link.
- **Password reset**: Custom `/forgot-password` (always 202, 3/min) and `/reset-password` (5/min) endpoints wrapping fastapi-users `UserManager`. Reset email with HMAC token link.
- **Session invalidation**: `password_changed_at` column + Alembic migration. Refresh endpoint rejects tokens issued before password change.
- **Frontend**: ForgotPasswordPage, ResetPasswordPage, "Forgot your password?" link on LoginPage, API functions.
- **Config**: Replaced 5 unused SMTP placeholder vars with `AWS_SES_REGION`, `AWS_SES_FROM_EMAIL`, `APP_URL`, `RESET_TOKEN_LIFETIME_SECONDS`.
- **Tests**: 60 new tests (17 email service + 3 auth password reset + 40 from Phase 6 already counted). 433 total passing.

## Decisions Made

- **Jinja2 over string.Template**: Autoescape prevents HTML injection from user-provided job/experiment/project names.
- **Explicit `RESET_TOKEN_LIFETIME_SECONDS`**: Don't rely on library defaults for security features.
- **`password_changed_at` check in refresh**: Closes stolen-cookie gap without a full token blacklist. Access tokens (30min) still survive — documented as known limitation for 8-10 trusted lab members.
- **Custom auth endpoints (not `get_reset_password_router()`)**: Matches existing login/register wrapper pattern, gives rate limiting control, prevents email enumeration.

## Open Items

- SES sandbox mode: New AWS accounts need production access request or verified recipient emails for testing.
- `AWS_SES_FROM_EMAIL` must be SES-verified in production.
- Access tokens (30-min TTL) not invalidated on password reset — acceptable for lab scale.
