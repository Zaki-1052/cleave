# 2026-03-24 — Phase 1.4: Manage Members + Notification Endpoints

## What was done

### Backend (4 files)
- Tightened `MemberCreate.role` and `MemberUpdate.role` to `Literal["admin", "contributor", "viewer"]` in `schemas/project.py` — Pydantic now auto-rejects invalid roles with 422
- Added duplicate membership check (`AlreadyMemberError`) and notification creation to `add_member()` in `services/project_service.py` — invited user gets a "Project Invitation" notification with inviter name and project name
- Added validation guards in `routers/projects.py`: self-role-change blocked (400), self-removal blocked (400), last-admin demotion/removal blocked (400), duplicate invite returns 409, list_members now requires project membership
- Implemented notification router endpoints (`GET /notifications`, `PATCH /notifications/:id/read`) — replaced 501 stubs with real service calls

### Frontend (4 files modified, 1 new)
- Added `addMember()`, `updateMemberRole()`, `removeMember()` API functions to `api/projects.ts`
- Added `useAddMember()`, `useUpdateMemberRole()`, `useRemoveMember()` mutation hooks with TanStack Query cache invalidation to `hooks/useProjects.ts`
- Created `components/projects/ManageMembersModal.tsx` — email + role invite form, member list with inline role dropdowns, own-role disabled, remove buttons, error handling for 404/409
- Wired modal into `pages/ProjectDetailPage.tsx` — admin-only "+ Manage Members" button opens modal

## Decisions made
- Used `Literal` type alias (`RoleType`) for role validation rather than manual checks — cleaner, Pydantic handles error messages
- Used native `<select>` with Tailwind styling instead of creating a reusable Select component (KISS — single consumer)
- Changed `add_member()` signature from `invited_by_id: int` to `inviter: User` — needed for notification message formatting
- Included notification endpoint implementation (from Phase 1.6 scope) since invite verification depends on it
- `notification_service.create_notification()` commits both the member and notification in one transaction via flush-then-commit pattern
