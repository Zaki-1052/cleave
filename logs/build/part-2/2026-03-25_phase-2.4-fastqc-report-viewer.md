# 2026-03-25 — Phase 2.4: FastQC Report Viewer Modal

## What Was Done

- Added `GET /experiments/{eid}/fastqs/{fid}/fastqc-summary` endpoint returning structured module pass/warn/fail data as JSON
- Added `find_fastqc_data_txt()` helper to resolve TXT data files from HTML report paths (handles real mode subdirectory and mock mode flat layouts)
- Fixed mock mode to copy TXT file alongside HTML so summary endpoint can parse it
- Added `FastqcModuleSummary` and `FastqcSummaryResponse` Pydantic schemas
- Created `FastqcReportModal` component: summary sidebar with status icons, iframe rendering HTML report, Download Report + Full Screen toolbar
- Replaced new-tab `<a>` link in FASTQs table FASTQC column with button that opens the modal
- Added `getFastqcSummary()` API function and `useFastqcSummary()` TanStack Query hook
- 6 new tests (3 unit for resolver, 1 mock TXT copy, 2 integration for summary endpoint), all 75 tests passing

## Decisions Made

- **Parse on-the-fly, not stored in DB**: Module summaries parsed from TXT file at request time. Avoids migration + backfill for a field only read when modal opens. Fine for 8-10 users.
- **Standalone modal, not extending Modal**: Base `Modal` is `max-w-2xl`; FastQC needs near-full-viewport. Follows `WizardModal` precedent for standalone full-viewport modals.
- **Iframe for HTML rendering**: FastQC HTML is self-contained (~900KB). Simplest approach with no DOM injection or CSS conflicts.
- **Same-origin fragment navigation**: `contentWindow.location.hash` for scrolling to modules, with cross-origin fallback via src swap.
