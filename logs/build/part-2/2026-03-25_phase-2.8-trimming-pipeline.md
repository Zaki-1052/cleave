### Step 3: Trimming Pipeline Module
- Created `backend/pipelines/trimming.py` — `TrimmingStage(PipelineStage)` with:
  - `validate()` — checks required params (experiment_id, project_id, fastq_pairs)
  - `run()` — two-stage: Trimmomatic PE (adapter+quality) → kseq_test (fixed-length 42bp)
  - `mock_run()` — copies input FASTQs to trimmed output paths (real files on disk per todos.md)
  - `generate_methods_text()` — manuscript-ready text with all parameter values
- Refactored `backend/pipelines/__init__.py` — stage registry dispatcher pattern
  - Registered pipelines use their stage's mock_run/run, unregistered types fall back to generic mock
  - Validation runs before execution

### Step 4: Worker Enhancement + Trimming Service
- Enhanced `backend/worker.py`:
  - Injects `job_id` into params before calling pipeline
  - Sets `started_at` timestamp when job begins
  - Post-pipeline dispatch: calls `create_trimmed_fastq_records()` for trimming jobs
  - Creates notifications (job_complete/job_error) for the launching user
  - Stores `methods_text` and `completed_at` on job completion
- Created `backend/services/trimming_service.py`:
  - Creates `FastqFile` records with `is_trimmed=True` for each trimmed FASTQ
  - Creates `JobOutput` records linking files to the job
  - Atomically updates `storage_bytes` on experiment and project
  - Triggers post-trim FastQC on trimmed files

## Decisions Made
- **Adapter detection threshold**: Uses FastQC's built-in warn/fail status (>5% adapters triggers warn) rather than custom parsing
- **No dedicated adapter endpoint**: Frontend derives banner state from existing FASTQ list data
- **Pipeline dispatcher refactored** to stage registry pattern — sets pattern for alignment/peak calling
- **Worker creates per-session DB connections** (not long-lived) for safety
- **Mock mode creates real files** — copies input FASTQs to trimmed paths so file browser/download work locally
